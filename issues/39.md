MVP scope for an initial Web UI focused on observability + safe previews.

## Goals
- Fast visibility into current state + last result
- Easy access to logs and run history
- Safe preview workflow via dry-run/plan against a ref/commit

## In scope (MVP)
- Dashboard: last run status, last/next schedule time, per-repo checkout SHA/ref and freshness (multi-repo aware)
- Runs history: list + details (start/end, duration, trigger source, outcome, summary counts, per-repo revisions)
- Logs: per-run logs + simple filtering (time/level/component) and copy-friendly output
- Plan / dry-run: run dry-run against a ref/commit (per-repo or all repos) and show a plan summary + diffs for changed quadlets with provenance info
- Managed units (read-only): list units quadsyncd manages + current systemd user status + last restart time when available

## Non-goals (MVP)
- Applying changes from the UI
- Mutating actions (restart units, change config) beyond triggering a dry-run
- Notifications and multi-host aggregation

## Notes / constraints
- Prefer: localhost bind + reverse-proxy for access; redact secrets in all views
- Prefer: plan-first UX (generate plan -> view -> copy/share), even if apply remains CLI-only

## Multi-repo considerations

> **Context:** Multi-repo support landed via #42, #46, #47. The codebase now supports `repository` (single) / `repositories` (list) config, priority-based conflict resolution (`internal/multirepo.Merge`), per-repo auth overrides (`GitClientFactory`), and multi-repo-aware webhook matching (`matchesConfiguredRepo`). The following adjustments ensure the Web UI plan accounts for these changes.

### Dashboard / overview (`GET /api/overview`)
- "Current checkout SHA/ref" is now **per-repository** (`State.Revisions map[string]string`). The overview response must return a `repositories[]` array with per-repo URL, ref, commit SHA, and freshness — not a flat single-value field.

### Run metadata (`meta.json`)
- Each run syncs **all** configured repos. `meta.json` must include:
  - `revisions: map[string]string` (repo URL → commit SHA, matching `State.Revisions`)
  - `conflicts: []Conflict` from the `multirepo.MergeResult` (so conflict resolution outcomes are visible in run history)
  - Per-repo file counts in `summary` where feasible.

### Plan / dry-run (`POST /api/plan`)
- The request body must support:
  - Planning against a **specific repo** (URL + ref/commit override) while keeping other repos at their current state.
  - Planning against **all repos** at their current or overridden refs.
- The plan response must include provenance (`source_repo`, `source_ref`, `source_sha`) on each diff item so the UI can show which repo each change originates from.
- Conflict resolution from `multirepo.Merge` must be part of the plan pipeline; plan output should surface any conflicts that would arise.

### Webhook route relocation
- When moving the handler from `POST /` to `POST /webhook`, preserve the current multi-repo-aware handler flow:
  - `GitClientFactory` pattern (not the old single `git.Client`)
  - `matchesConfiguredRepo` check
  - `repoFullNameFromURL` helper
  - `GitHubPushEvent.Repository.CloneURL`/`SSHURL` fields

### SSE event payloads
- Events (`run_started`, `run_updated`, `log_appended`, `plan_ready`) should include multi-repo context:
  - `run_started` / `run_updated`: include `repo_count`, per-repo URLs, and which repos had changes.
  - `plan_ready`: include provenance and conflict info.

### `performSync` instrumentation
- The serve-mode sync entry point (`Server.performSync`) now uses `sync.NewEngineWithFactory`. Run store instrumentation must wrap this factory-based flow, not the old single-client path.

## Implementation Plan (MVP)

### Architecture decisions (locked for MVP)
- Web UI lives inside `quadsyncd serve` (single process serves webhook + UI + API).
- Bind to localhost; production access via reverse proxy that provides auth.
- Webhook endpoint moves from `POST /` to `POST /webhook` to free `/` for UI.
- UI uses Vite + Svelte + TypeScript.
- Live updates via SSE (`EventSource`).
- Run history is cross-process: runs from `quadsyncd sync` (timer/CLI) and `quadsyncd serve` are both visible.
- Run history/logs/plans persisted as files under `paths.state_dir`.
- Retention: prune artifacts by age (default: 14 days).
- Plan/dry-run compares desired ref/commit against *current* `paths.quadlet_dir` (detect drift).
- Plan diffs are side-by-side; only quadlet file contents are diffed/rendered (do not expose companion file contents).
- Assume default systemd unit names exist for schedule display: `quadsyncd-sync.timer` / `quadsyncd-sync.service`.
- Web UI toolchain: Node 20 + pnpm.

### HTTP surface (routes)
- `POST /webhook` GitHub webhook (existing handler relocated; preserve `GitClientFactory` + `matchesConfiguredRepo` flow).
- `GET /` UI (SPA entry), `GET /assets/*` UI static assets.
- API (all JSON unless noted):
  - `GET /api/overview` dashboard aggregate (multi-repo: per-repo status in `repositories[]` array).
  - `GET /api/runs?limit=&cursor=` list runs (newest first).
  - `GET /api/runs/{id}` run metadata + summary (includes `revisions` map and `conflicts` list).
  - `GET /api/runs/{id}/logs?level=&component=&q=&since=` fetch logs (paged/seekable).
  - `GET /api/runs/{id}/plan` plan + diff metadata (if present; includes per-item provenance).
  - `POST /api/plan` trigger dry-run plan (body: `{repo_url?, ref?, commit?}` — omit `repo_url` to plan all repos).
  - `GET /api/units` managed units + systemd status.
  - `GET /api/timer` next/last trigger times for `quadsyncd-sync.timer` (best-effort).
  - `GET /api/events` SSE stream (`text/event-stream`).

### Run persistence format (files in `paths.state_dir`)
- Store each run under `runs/{runID}/`:
  - `meta.json` (start/end timestamps, trigger, per-repo revisions map, outcome, summary counts, conflict list, etc.).
  - `log.ndjson` (one JSON record per log line; sourced from `slog`).
  - `plan.json` + per-file diff artifacts (only for plans; only quadlet diffs; each diff item includes `source_repo`, `source_ref`, `source_sha`).
- Keep it append-only and resilient to partial writes (atomic temp+rename for metadata; logs append).
- Retention: prune runs older than 14 days on startup and periodically (and also from `sync` so history stays bounded even if `serve` is off).

### Back-end work items
1) Refactor `internal/webhook` to mount:
   - webhook handler at `/webhook` (preserve `GitClientFactory` + `matchesConfiguredRepo` + multi-repo handler flow from #47)
   - API handlers under `/api/*`
   - SSE broadcaster at `/api/events`
   - UI static assets at `/` and `/assets/*`
2) Add `internal/runstore` (filesystem-backed):
   - create/run lifecycle, write meta (including multi-repo `revisions` + `conflicts`), stream logs, read/query runs, retention prune.
3) Instrument `quadsyncd sync` (CLI/timer) to write run records + logs:
   - create per-run log file handler (ndjson) in addition to existing stdout logging.
   - capture per-repo revisions and merge conflicts in run metadata.
4) Instrument `quadsyncd serve` runs similarly:
   - initial sync on startup (uses `NewEngineWithFactory` — instrument this factory-based flow)
   - webhook-triggered sync (existing single-flight semantics remain)
   - UI-triggered plans
5) Implement planner for UI dry-run:
   - checkout requested ref/commit into isolated work dir (support per-repo or all-repos planning)
   - compute plan vs *current* `paths.quadlet_dir` (not vs previous state)
   - run `multirepo.Merge` as part of the plan pipeline; surface conflicts in plan output
   - deletes limited to managed set from `state.json` when `sync.prune=true`
   - generate side-by-side diff artifacts for quadlet extensions only; include provenance on each diff item
6) systemd read-only introspection:
   - managed units derived from `state.json` quadlets (note: `ManagedFile` now includes `source_repo`/`source_ref`/`source_sha` provenance)
   - status + timestamps via `systemctl --user` (best-effort)
   - timer next/last trigger via `systemctl --user show quadsyncd-sync.timer ...`
7) Security hardening:
   - keep webhook signature verification as-is (route-scoped: `/webhook` uses HMAC, `/api/*` uses CSRF)
   - add CSRF protection for UI POSTs (e.g. `POST /api/plan`)
   - set security headers (CSP, nosniff, referrer-policy, permissions-policy)
   - no CORS

### Frontend (Vite + Svelte + TS)
- Pages:
  - Dashboard (overview cards with per-repo status + last/next schedule + last run)
  - Runs list (filterable) + run detail (logs viewer + artifacts + per-repo revisions + conflict outcomes)
  - Plan view (select repo or all repos, enter ref/commit, trigger plan, view summary + side-by-side diffs with provenance badges)
  - Managed units (read-only; show source repo provenance where available)
- SSE integration:
  - single `EventSource('/api/events')`
  - update local stores on `run_started`, `run_updated`, `log_appended`, `plan_ready` (payloads include multi-repo context)
  - keep polling-free for MVP; fall back to refetch on reconnect

### Build & packaging
- `webui/` built with Vite; output embedded into Go binary via `//go:embed`.
- Use Node 20 + pnpm.
- CI/build approach to be implemented so `go build` has UI assets available (either generate during build or commit `dist/`; prefer generation + embed).

### Testing
- Unit tests:
  - `internal/runstore` (write/read/prune; multi-repo metadata round-trip)
  - planner (temp dirs: checkout vs quadlet_dir; add/update/delete; diff artifacts; multi-repo merge + conflict surfacing)
  - API endpoints via `httptest` (including SSE framing; overview multi-repo response shape)
  - CSRF enforcement on UI POST endpoints; webhook remains signature-based.

### Acceptance criteria
- `quadsyncd serve` serves UI at `/` and accepts GitHub webhooks at `POST /webhook`.
- UI shows last run + run history with logs from both CLI/timer and serve/webhook runs.
- Dashboard displays per-repo status (SHA, ref, freshness) for all configured repositories.
- Run detail view shows per-repo revisions and any conflict resolution outcomes.
- UI can trigger a dry-run plan (per-repo or all repos) for a ref/commit and render side-by-side diffs with provenance info for quadlet files.
- Managed units list shows systemd user status best-effort.
- Retention prunes runs older than 14 days.
- No secrets/config contents are exposed; companion file contents are not diffed/rendered.
